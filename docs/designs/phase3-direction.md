## Vision

### 10x Check
Phase 3 将海龟汤从"单机答题 App"变成"轻社区推理平台"：
- 每天打开是因为 **Streak 断了会可惜**（日历 + 连签奖励）
- 每道题玩完都会看 **其他人的评分和评价**（理解自己的水平）
- 排行榜不是炫耀，是 **社交压力引擎**（朋友看得到你今天第几）
- Profile 页的徽章墙是 **身份认同**，不是统计数字

### Platonic Ideal
打开 App，首页右上角显示"🔥 连续第 7 天"。每日谜题旁边有排行小卡（前 3 名头像 + 用时）。
玩完一局，结果页出现"新徽章：推理侦探 🕵️（赢了 10 局！）"的弹窗。
个人主页像一本"推理日记"：日历热力图、徽章墙、历史对话可折叠查看。

## Architecture Notes

### 新数据模型

```
UserStreak {
  userId        Int
  currentStreak Int       // 当前连签天数
  longestStreak Int       // 历史最长
  lastPlayDate  Date      // 上次游玩日期（UTC date）
  createdAt     DateTime
  updatedAt     DateTime
}

DailyLeaderboard {
  puzzleId   Int          // 当日谜题 ID
  userId     Int
  date       Date         // 日期分区键
  durationSec Int
  questionCount Int
  status     SessionStatus // WON / GIVEN_UP（注：schema 枚举是 GIVEN_UP 不是 GAVE_UP）
  rank       Int?         // 不存储；查询时用 ROW_NUMBER() OVER 实时计算，初期够用
  createdAt  DateTime
}

UserBadge {
  // 不存 DB，全部在 /profile API 通过 stats 计算返回
  // 避免额外 migration，纯计算 badges 数组
}
```

### 新 API 端点（草案）

```
GET  /api/v1/profile/streak          → {currentStreak, longestStreak, lastPlayDate, calendar[]}
GET  /api/v1/puzzles/daily/leaderboard → [{rank, nickname, durationSec, questionCount}]  // 不返回 userId（安全）
GET  /api/v1/profile/badges          → [{id, name, emoji, description, earned, earnedAt?}]
POST /api/v1/games/:id/streak-checkin → 游戏结束时触发，更新 streak（幂等）
```

## Key Implementation Risks

1. **Streak 时区问题（已决策）**：采用"UTC day"策略——`lastPlayDate` 存储 UTC 日期字符串（`YYYY-MM-DD`，即 `new Date().toISOString().slice(0,10)`）。`streak-checkin` API 无需客户端传 timezone，服务端统一用 UTC day 判断"今天"。理由：用户分布在同一地区（中国，UTC+8），UTC day 与本地 day 差距最多1天，且实现简单无状态。边界情况：中国时区 23:59-00:59 打游戏会跨 UTC day，此时 streak 判断可能偏差1天——这是可接受的取舍。
2. **排行榜实时性 vs 性能**：实时 `rank()` OVER PARTITION 对高并发有压力。初版可 5 分钟缓存刷新（Redis 或内存缓存），上线后再观察。
3. **徽章评定一致性**：`/profile/badges` 每次实时计算，需确保 stats 字段（won_games、hint_used 等）是准确的累计值，不能有漂移。
4. **每日谜题与排行榜强耦合**：`date-hash` 轮换逻辑必须与排行榜的 `date` 分区键对齐，否则排行榜会跨日混乱。
